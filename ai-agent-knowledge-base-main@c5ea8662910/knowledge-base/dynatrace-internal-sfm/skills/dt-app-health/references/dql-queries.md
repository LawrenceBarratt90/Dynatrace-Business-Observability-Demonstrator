# DQL Query Apps footprint

Investigate DQL query failures and billed data consumption generated by a Dynatrace app.

All queries use bucket: `custom_sen_low_query_frontend_dql_logs` with `scanLimitGBytes:-1`.

## When to Use

- An app is hitting DQL query errors or timeouts seen by users
- Investigating unexpectedly high Grail query consumption costs
- Identifying which feature/component within an app is responsible for expensive queries

## Query Failures

Top 10 failing DQL queries grouped by source and error type, with per-user failure frequency.

```dql
fetch logs, bucket:{"custom_sen_low_query_frontend_dql_logs"}, scanLimitGBytes:-1
| filter isNotNull(app)
| filter state != "succeeded"

| parse content, "LD JSON:GrailLog"
| filter isNotNull(GrailLog[error])
| fieldsAdd error = GrailLog[error]

| parse GrailLog[client.client_context], "JSON:JsonClientContext"

| fieldsAdd Source = coalesce(
    GrailLog[client.function_context],
    GrailLog[client.internal_service_context],
    JsonClientContext[source],
    JsonClientContext[package],
    GrailLog[client.client_context],
    GrailLog[client.application_context]
)

| makeTimeSeries {
  Users = countDistinct(user, default:0),
  Count = count(default: 0)
}, by: {Source, error}, bins: 30

| sort arraySum(Count) desc
| limit 10

| fieldsAdd times = concat("Per user: x", round(arraySum(Count)/arraySum(Users), decimals:1))
```

**`Source` field:** Resolved from client context fields in priority order — gives the feature/package/component that issued the query. Useful for identifying which part of the app is failing.

**`state != "succeeded"`**: Captures all non-successful outcomes — errors, timeouts, cancelled queries.

## Billed Consumption

Top 20 query sources by total billed bytes (queries consuming > 1 MB), with average consumption per query and estimated list-price cost.

```dql
fetch logs, bucket:{"custom_sen_low_query_frontend_dql_logs"}, scanLimitGBytes:-1
| filter billable_bytes > 0

| parse content, "LD JSON:GrailLog"
| parse GrailLog[client.client_context], "JSON:JsonClientContext"

| fieldsAdd Source = coalesce(
    GrailLog[client.function_context],
    GrailLog[client.internal_service_context],
    JsonClientContext[source],
    JsonClientContext[package],
    GrailLog[client.client_context],
    GrailLog[client.application_context]
)

| makeTimeSeries {
  count = count(),
  consumedSum = sum(billable_bytes)
}, by: {Source}, bins: 30

| filter arraySum(consumedSum) > 1024*1024 // 1 MB minimum
| sort arraySum(consumedSum) desc
| limit 20
| sort arraySum(consumedSum) asc

| fieldsAdd consumptionPerQuery = consumedSum[]/count[]
| fieldsAdd pricePerQuery = consumptionPerQuery[]/1e+9*0.0035

| fieldsAdd avgSizeFormatted = concat(round(arrayAvg(consumptionPerQuery)/1e+9, decimals: 2), " GB / per query")
```

**`pricePerQuery`**: Estimated cost at $0.0035 per GB (Grail query list price). Use for relative comparison only — actual billing depends on contracted rates.

**`consumedSum`**: Total billed bytes across all queries from that source within the time series. Filter `> 1024*1024` removes very small/test queries.

## Scoping to a Specific App

The failures query uses `| filter isNotNull(app)` and the consumption query uses `| filter billable_bytes > 0` as base filters. To scope either query to a single app, add:

```dql
| filter app == $app
```

Insert this line after the initial `filter` line. The `app` field in the DQL log contains the app ID that issued the query.

---

## `Source` Field Resolution

The `Source` field is a best-effort resolution of which feature or component issued the query, resolved from client context in this priority order:

| Priority | Field | Description |
|---|---|---|
| 1 | `client.function_context` | Specific function/handler name |
| 2 | `client.internal_service_context` | Internal service calling Grail |
| 3 | `JsonClientContext[source]` | Source field from parsed client context JSON |
| 4 | `JsonClientContext[package]` | Package name from parsed client context JSON |
| 5 | `client.client_context` | Raw client context string |
| 6 | `client.application_context` | Application-level context |

---

## Notes

- **Always use `scanLimitGBytes:-1`** — DQL log queries scan large volumes and hit limits without this setting
- `billable_bytes` is only present on queries that scanned data; queries that returned from cache may have `0` or null
- Timeframe: default to the last 1–2 hours for real-time failure investigation; extend to 24h for cost analysis
- If `Source` is null for many rows, the app may not be setting client context — this is a gap in observability
