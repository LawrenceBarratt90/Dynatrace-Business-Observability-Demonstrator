# kb-run Coding Conventions

TypeScript ESM CLI tool for building AI agent evaluation environments.

## Build Commands

```bash
npm install          # install dependencies
npm run build        # compile TypeScript (tsc) + copy packages to dist/
npm run dev          # watch mode (tsc --watch)
npm link             # install globally as `kb-run` command
```

## Code Style

### TypeScript / ESM

- ESM modules throughout: `"type": "module"` in package.json
- Use `.js` extensions in all relative imports (required by ESM resolution):
  ```typescript
  import { scaffold } from './scaffold.js';
  import type { BuildOptions } from '../types.js';
  ```
- TypeScript strict mode enabled (`"strict": true` in tsconfig.json)
- Target ES2020, module ESNext, Node module resolution

### Patterns

- `commander` for CLI commands; register each command in a separate file under `src/commands/`
- `chalk` for colored console output
- `fs-extra` for file operations, `fs/promises` for native async fs
- `@inquirer/prompts` for interactive CLI prompts
- ESM `__dirname` equivalent:
  ```typescript
  const __filename = fileURLToPath(import.meta.url);
  const __dirname = dirname(__filename);
  ```

### Documentation

- JSDoc comments on all exported functions with `@param`, `@returns`, `@throws`, `@example`
- Module-level JSDoc block at the top of each file explaining purpose
- Inline comments for non-obvious logic

### Error Handling

- Throw `Error` with descriptive messages including context (paths, values)
- Top-level command actions catch errors, print with `chalk.red`, and `process.exit(1)`
- Validate inputs early (fail fast) before doing work
- Use `async/await` consistently; no raw promise chains

### Naming

- Files: kebab-case (`command-builder.ts`, `validate.ts`)
- Types/interfaces: PascalCase (`BuildOptions`, `TargetIDE`)
- Functions: camelCase (`registerBuildCommand`, `validateInputFiles`)
- Constants: UPPER_SNAKE_CASE for true constants, camelCase for derived values

## aimgr Integration

kb-run **must always use its own internal aimgr repository** â€” never the global user config.

- Every `aimgr` invocation must pass `--config <path>` where `<path>` is the runtime-generated `.aimgr-config-resolved.yaml` inside the kb-run package root.
- This config points the aimgr repo at a local `.aimgr-repo/` directory and the local knowledge-base source. It contains **no remote sources and no GitHub URLs**.
- The config is generated by `getResolvedAimgrConfig(kbRoot)` in `src/aimgr.ts`. This function is the **single source of truth** â€” import and call it everywhere an aimgr command is needed.
- The global `~/.config/aimgr/aimgr.yaml` must **never** be read by kb-run. A missing or empty global config must not break anything.
- Use `spawnSync` with an args array (not `execSync` with a shell string) for all aimgr calls to avoid shell-quoting issues on Windows.

## Source Structure

```
src/
  index.ts              # CLI entry point (commander setup)
  commands/
    build.ts            # Build command implementation
    start.ts            # Start/run command implementation
  scaffold.ts           # Directory scaffolding and file generation
  validate.ts           # Input validation (KB root, dtctl, packages)
  interactive.ts        # Interactive prompt flows
  command-builder.ts    # Command string construction
  aimgr.ts              # aimgr integration
  types.ts              # Shared type definitions
  runtime/              # Runtime modules and runtime contract helpers
```

runtime/
  system-bash.yaml      # Runtime-owned resources for native bash mode
  py-bash.yaml          # Runtime-owned resources + py-bash config
  dt-mcp-server.yaml    # Runtime-owned resources + generic MCP config
```

## Runtime Model

- Runtime is an explicit first-class input: `system-bash`, `py-bash`, or `dt-mcp-server`
- Add new runtime behavior under `src/runtime/` instead of adding ad-hoc branches in `scaffold.ts`
- Runtime YAML files live under `runtime/<runtime>.yaml`
- Every runtime YAML must define `resources:` for arbitrary aimgr resources
- Runtime-specific sections are optional and should only exist when the runtime has real configurable inputs
  - `system-bash.yaml` should usually contain only `resources:`
  - `py-bash.yaml` may define `py-bash:` values such as `repoPath`
  - `dt-mcp-server.yaml` may define `dt-mcp-server:` values such as generic MCP server config
- Runtime YAML stays client-agnostic; target-specific rendering belongs in the runtime module
- All runtime YAML string values support env interpolation:
  - `${VAR_NAME}` â†’ required, fail if missing
  - `${VAR_NAME:-default}` â†’ optional with fallback
